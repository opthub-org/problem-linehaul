# 幹線輸送 Next-Hop 最適化コンペ 説明資料

## 1. 概要

本コンペでは、日本全国 46 都道府県を対象とした幹線輸送ネットワークの設計問題に取り組みます。

参加者には、都道府県ごとの情報、都道府県間の荷量、都道府県間の時間・距離、費用定義、および解の評価プログラムが与えられます。参加者は、業務上の制約である `next-hop` 形式に従って、各都道府県から各着都道府県向け荷物を次にどの都道府県へ渡すかを決定し、最も低コストとなる解を出力するプログラムを作成してください。

## 2. 課題設定

### 2.1 対象

* 対象地域は沖縄県を除く 46 都道府県です。
* 各都道府県は 1 つの拠点を持つものとします。
* 拠点間はトラック幹線輸送で接続されます。

### 2.2 入力として与えられるもの

* 都道府県の情報(prefectures.csv)
* 都道府県間の平均荷量(virtual_prefecture_flow.csv)
* 幹線輸送の際の時間・距離(truck_distance_time_long.csv)

### 2.3 参加者が解く問題

各 `発都道府県` と `着都道府県` の組に対して、

* 現在その荷物を持っている都道府県から
* 次にどの都道府県へ荷物を送るか

を `next-hop` として決定してください。

ここで `発都道府県` は元の出荷地だけではなく、中継後に荷物を持っている都道府県も含みます。

## 3. Next-Hop の意味

輸配送の現場では荷物を向け地ごとに中継拠点に1箇所に集約して配送を行うことが多いです。
また、現場での混乱を避けるために、ある場所へ送付する際の中継拠点は頻繫に変更されません。

ここで`next-hop` は以下を意味します。

* 「発都道府県 `o`から 着都道府県 `d` 向けの荷物を送付する際に、次に送る 都道府県は `next-hop` である」

したがって、出力ファイルは配送経路そのものを全列挙するのではなく、各 `(現在地, 最終着地)` に対して次に送付する都道府県だけを指定する形になります。

直送が最適な場合は、`next-hop = 着都道府県` になります。

## 4. 入力データ


### 4.1 都道府県情報

* `prefecture_code`
* `prefecture_name`
* `latitude`
* `longitude`

### 4.2 都道府県間荷量

* `origin_prefecture_code`
* `destination_prefecture_code`
* `virtual_load`
* `standard_deviation`

`virtual_load` は平均荷量、`standard_deviation` は荷量変動の標準偏差です。

各区間の荷量は独立として、複数区間の荷量を集約した場合には、平均の合計、分散の合計で表されるものとします。

### 4.3 都道府県間時間・距離

例:

* `origin_prefecture_code`
* `destination_prefecture_code`
* `distance_km`
* `time_min`

### 4.4 コスト定義

評価時の基本的な考え方は以下です。

* 車両は時間に比例して費用が発生する(10000円/時間)
* 最大8 時間の距離まで輸配送可能である。 
  * したがって 1 台あたりの区間費用は `10000 × time_min / 60`
* 1 台の最大積載量は 100,000 kgとする
* 区間ごとの荷量平均と標準偏差から、必要車両数の期待値を計算して評価する
  * 荷量のブレは正規分布を仮定するものとする
  * 必要車両数の期待値の計算は別途提供されるPythonのコードを参照

## 5. 業務上の制約

本コンペでは、以下を必須条件とします。

### 5.1 Next-Hop 制約

出力は必ず `next-hop` 形式であること。

### 5.2 一意な次拠点

各 `(現在地, 着地)` に対して、次に送る都道府県は 1 つでなければなりません。

### 5.3 到達可能性

`next-hop` をたどったとき、必ず最終的に着都道府県へ到達しなければなりません。

### 5.4 ループ禁止

`next-hop` をたどる過程で循環があってはいけません。

### 5.5 実在アークのみ利用

出力された `next-hop` は、入力として与えられた都道府県間時間・距離データ上で有効な移動でなければなりません。

## 6. 解の形式

### 6.1 解の形式
参加者は、以下の2116要素(=46*46)の1次元配列をJSON形式で入力してください。

$[\ a_{1\ 1}, a_{1\ 2}, ..., a_{1\ 46}, a_{2\ 1},..., a_{2\ 46}, a_{46\ 1},... ,a_{46\ 46}\ ]$

ここで$a_{i\ j}$は$i$番目の都道府県から$j$番目の都道府県に送る際に次に送る都道府県のprefecture_code(int)です。

例）1:北海道から13:東京へ送る際に、次に2:青森へ送付する場合

$a_{1\ 13} = 2$

また同じ都道府県に送る場合は入力によらずどの都道府県をコストは0となります。

（評価システムの都合上、整数値を入力する必要があるため、$a_{i\ i}=i$としてください。(自身のprefecture_codeにする。）)

## 7. 評価方法

主催者から、`next-hop` 解を評価するプログラムを提供します。

評価プログラムは以下を行います。

1. 各 OD 荷物について、`next-hop` を終点までたどって経路を復元する
2. 各区間に流れる平均荷量と標準偏差を集計する
3. 区間ごとの `expected_vehicle_count` を算出する
4. 区間ごとの費用を計算する
5. 全区間費用の合計を解のスコアとする

スコアは小さいほど良いものとします。

### 7.1 実行方法
1. linhaul-problem\main.pyを以下の実行時引数で起動してください。 
>  "-c problem" "-t problem"
2. input?と表示されるので、解(2216個の整数のリスト)を入力する。
> input? 
 
>[ 1, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2, 2,...,2]
3. 入力が適切であれば以下の様に計算結果が表示される。
> {"objective": 211351547.4309249, "feasible": true, "constraint": null}
* objective：入力に対しての目的関数値
* feasible：実行成功時にTrueになる
* constraint：Null(入力によらずNullになります)


## 8. 評価の考え方

この問題では、単純に総走行距離を短くすればよいわけではありません。

* 複数荷物をうまく集約すると、期待車両数が下がる可能性があります
* 一方で、遠回りをすると区間時間コストが増えます
* `next-hop` 制約があるため、OD ごとに自由な経路を個別に選ぶ問題ではありません
* ある着地に対して、各現在地で一意な次拠点を決める必要があります

そのため、

* ハブ集約
* 方面別の幹線共有
* 期待車両数を意識した統合

などが重要になります。
